Every `/aif-fix` leaves behind a **patch** — a structured record of what went wrong and how it was fixed. `/aif-evolve` processes patches incrementally, analyzes your project's stack and conventions, and turns recurring patterns into **rules** that make skills smarter.

## The learning loop

```text
/aif-fix -> patches -> /aif-evolve -> skill-context -> better /aif-fix, /aif-implement, /aif-review
```

Each pass makes the AI more aware of your project's real failure modes. After enough fixes, evolution stops being generic advice and starts producing project-specific guards.

Incremental state is tracked in:

```text
<paths.evolutions>/patch-cursor.json
```

- No cursor: first run reads all patches
- Cursor present: later runs read only new patches plus a small overlap window
- Full rescan: delete the cursor and rerun `/aif-evolve`

## Skill-context

Built-in `aif-*` skills are overwritten on package update. To keep project-specific rules stable, `/aif-evolve` writes them to a separate fixed location:

```text
.ai-factory/skill-context/<skill-name>/SKILL.md
```

Key properties:

- **Survives `ai-factory update`** — lives in your repository
- **Higher priority than base rules** — project-specific context wins on conflict
- **Cumulative** — evolve adds, updates, and removes rules over time

### Example

```md
# Project Rules for /aif-fix

> Auto-generated by `/aif-evolve`. Do not edit manually.
> Last updated: 2026-02-15 14:30
> Based on: 12 analyzed patches

## Rules

### Null-check for optional relations
**Source**: 2026-02-10-09.15.md, 2026-02-12-16.45.md
**Rule**: Check for optional or nullable fields before accessing nested properties.
```

## Running `/aif-evolve`

```text
/aif-evolve
/aif-evolve fix
/aif-evolve implement
/aif-evolve all
```

Skill names are normalized automatically: `fix` becomes `aif-fix`, `/aif-plan` becomes `aif-plan`.

## Workflow overview

| Step | What happens |
| --- | --- |
| `0` | Resolve target skill and load project context |
| `1` | Read patches incrementally and extract prevention points |
| `2` | Read target base SKILL files |
| `3` | Compare skill-context rules against upstream skill rules |
| `4` | Present stale-rule findings to the user |
| `5` | Analyze actual gaps not covered by base skill or skill-context |
| `6` | Generate concrete improvements linked to patches or conventions |
| `7` | Apply approved changes and write evolution logs |
| `8` | Suggest next actions such as `/aif-review` or new skill generation |

## Built-in vs custom skills

| Skill type | Where rules are written | Why |
| --- | --- | --- |
| Built-in `aif-*` | `.ai-factory/skill-context/<skill>/SKILL.md` | base skill files are overwritten on update |
| Custom project skills | `skills/<skill>/SKILL.md` | you own the file directly |

## Stale rule cleanup

When AI Factory is updated, upstream SKILL files may start covering rules that previously existed only in your skill-context. `/aif-evolve` handles four cases:

- **Fully covered** — upstream rule now includes your behavior
- **Contradiction** — upstream and project rule conflict
- **Partial overlap** — both cover part of the same behavior
- **Still unique** — rule remains project-specific

Only skill-context files are changed during cleanup. Base package skills are never edited by evolve.

## When to evolve

- After `5` to `10` fixes
- After changing tech stack or major libraries
- When the same bug type keeps recurring
- After `ai-factory update`, to clean stale project overrides
